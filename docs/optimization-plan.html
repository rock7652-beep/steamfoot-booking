2 DB Schema 調整建議

根據需求，Schema 需要以下變更。已完成的標記綠色。

2-A. Customer Model 變更

prisma/schema.prisma — Customer
model Customer {
  id                 String        @id @default(cuid())
  userId             String?       @unique
  name               String
  phone              String
  email              String?       @unique
  googleId           String?       @unique
  avatar             String?       // ✅ 已完成 — Google 大頭貼
  authSource         AuthSource    @default(MANUAL) // ✅ 已完成 — 來源
  lineName           String?
  notes              String?
  assignedStaffId    String?       // ✅ 已 nullable — 建立時不需選店長
  ...
}
不需要額外改動：assignedStaffId 在原 schema 中已經是 nullable（String?），建立顧客時不選店長的需求已經被滿足。目前 createCustomer action 也已支援不傳 assignedStaffId。

2-B. StaffPermission — 已有 RBAC 基礎，無需改動

prisma/schema.prisma — StaffPermission（現有，不改動）
model StaffPermission {
  id         String  @id @default(cuid())
  staffId    String
  permission String  // "customer.read", "booking.create", etc.
  granted    Boolean @default(true)
  @@unique([staffId, permission])
}

Schema 層面已支援 RBAC，但目前 actions/queries 裡大多只檢查 role（OWNER/MANAGER），需要改為檢查 StaffPermission。

2-C. Schema 總結 — 無須新增 Migration

Model變更項目狀態
Customer新增 avatar, authSource已完成
CustomerassignedStaffId nullable原本就是
StaffPermissionRBAC key-value 權限表原本就有
AuthSource enumGOOGLE / EMAIL / MANUAL已完成
結論：DB Schema 不需要再跑 migration。所有需求都可以透過修改 application 層（UI + actions + queries）達成。

3-B. 權限檢查邏輯

權限判斷流程： Request 進入 │ ▼ 是否已登入？ ──NO──▶ 401 Unauthorized │ YES ▼ Role = OWNER？ ──YES──▶ ✅ 全部放行（Owner 擁有所有權限） │ NO ▼ Role = MANAGER？ │ YES ▼ 查詢 StaffPermission 表 │ ▼ permission.granted = true？ ──NO──▶ 403 Forbidden │ YES ▼ ✅ 放行 資料範圍規則（Data Scope）： 「customer.read」 → 所有店長都能看到全部顧客（共同服務） 「transaction.*」 → 營收歸屬僅記錄直屬店長（revenueStaffId） 「report.read」 → Owner 看全店，Manager 只看自己的營收

3-C. 主要改動點

目前 actions/queries 裡的權限檢查大多是 requireStaffSession()（只檢查 OWNER 或 MANAGER），需改為使用 checkPermission()。

改動前 vs 改動後
// ❌ 目前：只區分 role
const user = await requireStaffSession();
if (user.role !== "OWNER") throw new Error("無權限");

// ✅ 改後：使用細粒度 permission
const user = await requireStaffSession();
await checkPermission(user.role, user.staffId, "wallet.adjust");

4-C. 報表 Query 重構

現有的 monthlyStoreSummary 和 monthlyRevenueByCategory 只接受 year + month 參數。需要改為接收通用日期範圍。

報表 Query 新介面
// 新的通用日期範圍型別
type ReportDateRange = 
  | { mode: "day";     date: string }                    // "2026-04-04"
  | { mode: "range";   startDate: string; endDate: string } // 自訂區間
  | { mode: "month";   year: number; month: number }     // 2026-04
  | { mode: "quarter"; year: number; quarter: 1|2|3|4 }  // 2026-Q1

// 轉換為 startDate / endDate 的 helper
function resolveRange(range: ReportDateRange): { start: Date; end: Date }

// 報表查詢統一改為接收 ReportDateRange
async function storeSummary(range: ReportDateRange) { ... }
async function revenueByCategory(range: ReportDateRange) { ... }

資料可見性矩陣： ┌──────────────────┬──────────────┬──────────────┬──────────────┐ │ 資料類別 │ Owner │ Manager │ Customer │ ├──────────────────┼──────────────┼──────────────┼──────────────┤ │ 顧客基本資料 │ 全部 │ 全部 │ 僅自己 │ │ 顧客預約紀錄 │ 全部 │ 全部 │ 僅自己 │ │ 消費/儲值紀錄 │ 全部 │ 全部（僅查看） │ 僅自己 │ │ 營收歸屬 │ 全店 │ 自己的客戶 │ — │ │ 報表 │ 全店 + 各店長 │ 僅自己數據 │ — │ └──────────────────┴──────────────┴──────────────┴──────────────┘ 核心原則： 「看」可以共享（共同服務），「歸屬」只記直屬（營收分析）

5 實作順序

建議分 4 個 Phase，每個 Phase 可獨立部署上線。

Phase 1 — Layout 重構（預估 1~2 天）

左側 Sidebar + 全域 Layout

  • 建立 Sidebar 元件（展開/收合 + localStorage 記憶）
  • 建立 DashboardLayout 共用 layout，移除原有 bottom nav（桌面）
  • Mobile: 保留 bottom quick nav，加 hamburger 觸發 Sidebar overlay
  • Sidebar 項目根據 session.role + permissions 動態顯示
Phase 2 — RBAC 權限落地（預估 1 天）

細粒度權限檢查

  • 新增 wallet.adjust 和 staff.manage 到 ALL_PERMISSIONS
  • 所有 server actions 從 requireOwnerSession → checkPermission
  • Sidebar + 頁面按鈕 根據權限條件渲染
  • Owner 的「員工管理」頁面加入權限開關 UI（toggle 每個 permission）
Phase 3 — 報表強化（預估 1 天）

多維度日期篩選

  • 建立 ReportDateRange 型別 + resolveRange helper
  • 重構 storeSummary / revenueByCategory query
  • 報表頁 UI: 日期模式切換（單日/區間/月/季）
  • Manager 的報表自動限制為自己的營收數據
Phase 4 — UX 細節優化（預估 0.5 天）

Autocomplete + 顧客列表微調

  • customer-search.tsx: 強化 debounce、鍵盤操作、顯示更多欄位
  • 顧客列表: 加「來源」欄位（authSource badge）
  • 顧客列表: 加「直屬店長」篩選 dropdown
  • 確認 createCustomer 不強制 assignedStaffId
建議：Phase 1 → 2 → 4 → 3 的順序也可以，取決於是先讓日常操作更順暢（Layout + RBAC + UX），還是先讓報表功能更完整。如果店長已經在使用系統，建議先做 Phase 1 + 2 + 4 讓體驗升級，報表最後補。
