1 整體架構設計

1-A. 架構選型：API Bridge 模式

不合併 DB，而是在身體數據系統建立一組 API endpoint，讓蒸足系統可以安全地讀取資料。

┌─────────────────────┐ ┌─────────────────────┐ │ 蒸足預約系統 │ │ 身體數據系統 │ │ (booking-system) │ │ (health-tracker) │ │ │ │ │ │ Prisma + Postgres │ │ Supabase Client │ │ NextAuth v5 │ │ Supabase Auth │ │ │ │ │ │ ┌───────────────┐ │ HTTPS │ ┌───────────────┐ │ │ │ Health Data │──┼────────▶│ │ /api/health │ │ │ │ Service │ │ API Key │ │ (新建 API) │ │ │ │ (讀取＋快取) │◀─┼────────│ │ │ │ │ └───────────────┘ │ │ └───────┬───────┘ │ │ │ │ │ │ │ │ ▼ │ │ ▼ │ │ ┌───────────────┐ │ │ ┌───────────────┐ │ │ │ Customer 頁面 │ │ │ │ body_records │ │ │ │ 身體數據摘要 │ │ │ │ profiles │ │ │ └───────────────┘ │ │ └───────────────┘ │ └─────────────────────┘ └─────────────────────┘ 資料流向：單向讀取（蒸足 ← 身體數據） 對應機制：email 為主鍵比對 認證方式：API Key（server-to-server）

1-B. 為什麼不合併 DB？

方案優點缺點建議
合併到同一 Supabase 直接 JOIN，簡單 遷移風險高、兩套 Auth 衝突、RLS 複雜 不建議
API Bridge（推薦） 零遷移、各自獨立部署、安全 需建 API、有網路延遲 推薦
DB Replica / Foreign Data 接近原生 JOIN Supabase 不支援 FDW、維運複雜 不適用

2 顧客對應機制設計

2-A. 對應策略：Email 為主鍵

兩套系統都有 Google OAuth，Google 帳號的 email 是最穩定的共用識別。

對應流程： 蒸足 Customer 身體數據 profiles ┌──────────────────┐ ┌──────────────────┐ │ id: cuid_abc123 │ │ id: uuid-xyz-789 │ │ email: a@g.com │─── email 比對 ──│ email: a@g.com │ │ googleId: 111... │ │ (auth.users) │ │ name: 王小明 │ │ full_name: 王小明 │ │ phone: 0912... │ │ phone: 0912... │ │ healthProfileId │──────────▶│ id: uuid-xyz-789 │ └──────────────────┘ └──────────────────┘ 新增欄位（蒸足 Customer）： healthProfileId String? // 身體數據系統的 profile UUID healthSyncedAt DateTime? // 最後同步時間

2-B. 綁定時機與方式

方式觸發時機說明
自動比對 店長開啟顧客詳情頁 以 Customer.email 呼叫身體數據 API 查詢 profile，若找到則自動綁定 healthProfileId
手動綁定 店長點「連結身體數據」 輸入 email 或手機號碼搜尋身體數據系統，確認後綁定
批量比對 後台管理工具 一次比對所有 Customer email，顯示匹配結果供確認
建議第一階段用「自動比對」：店長點進顧客頁時，背景自動以 email 查詢。找到就顯示身體數據摘要，找不到就顯示「尚未連結」。零額外操作成本。

3-B. 安全機制

API 認證：Server-to-Server API Key 蒸足系統（後端） 身體數據系統 ┌──────────────┐ ┌──────────────────┐ │ Server Action │── X-API-Key ──▶│ /api/health/* │ │ or Query │ │ │ │ │ │ 驗證 API Key │ │ 不經過前端 │ │ ✅ → 回傳資料 │ │ 不暴露 Key │ │ ❌ → 401 │ └──────────────┘ └──────────────────┘ 安全要點： 1. API Key 存在蒸足系統的 .env（HEALTH_API_KEY） 2. 身體數據系統的 .env 也存一份用於驗證（INTERNAL_API_KEY） 3. API 只開放 GET（唯讀），不開放寫入 4. Rate limit: 100 req/min（防濫用） 5. 只回傳必要欄位（不回傳 auth 資訊）

4-B. 經營提示邏輯

條件提示內容顏色
最後量測 ≤ 14 天近期有量測紀錄（活躍顧客）綠
最後量測 15~30 天已超過 2 週未量測，建議提醒回訪橙
最後量測 > 30 天已超過 1 個月未量測，可能流失紅
體重/體脂持續下降療程見效，趨勢良好綠
任何指標 status=warning顯示具體警示，可推薦加強課程橙
totalRecords < 3紀錄較少，建議鼓勵持續追蹤橙

5 分階段實作計畫

Phase 1 — API 建立 + 基礎連結（預估 1~2 天）

在身體數據系統建 API，在蒸足系統做基礎讀取

  • 身體數據系統：新增 /api/health/profile、/api/health/summary 兩個 API route
  • 身體數據系統：API Key 驗證 middleware
  • 蒸足系統：Customer schema 新增 healthProfileId、healthSyncedAt
  • 蒸足系統：建立 HealthDataService（呼叫 API + 快取 5 分鐘）
  • 蒸足系統：顧客詳情頁加入「身體數據摘要」區塊（最新一筆 + 警示）
  • 蒸足系統：自動比對邏輯（開啟顧客頁時以 email 查詢 + 綁定）
Phase 2 — 趨勢圖表 + 經營提示（預估 1 天）

加入視覺化與經營輔助工具

  • 身體數據系統：新增 /api/health/records（歷史紀錄）
  • 蒸足系統：顧客頁加入 Recharts 趨勢圖（體重 + 體脂折線圖）
  • 蒸足系統：經營提示引擎（根據量測頻率、趨勢、警示生成提示）
  • 蒸足系統：顧客列表加入「最近量測」欄位 + 排序
Phase 3 — 批量同步 + 通知整合（預估 1 天）

自動化對應與回訪提醒

  • 蒸足系統：後台「批量比對」工具（一鍵比對所有 Customer email）
  • 蒸足系統：定時同步排程（每日凌晨更新所有已連結顧客的摘要快取）
  • 蒸足系統：回訪提醒列表（超過 N 天未量測的顧客清單）
  • 通知預留：搭配 customer.email + booking 系統發送提醒 email
Phase 4（未來）— SSO + 深度整合

統一登入與雙向資料流

  • 統一 Google OAuth（兩套系統共用 Google Client ID）
  • 身體數據系統的「預約教練」按鈕直接跳轉蒸足預約頁

7 風險評估與備案

風險影響緩解措施
Email 不一致（兩邊不同 email） 自動比對失敗 提供手動搜尋綁定功能（以 phone 或 name 輔助比對）
身體數據 API 不穩定 / Vercel 冷啟動 顧客頁載入慢 5 分鐘快取 + 非阻塞載入（先顯示蒸足資料，身體數據異步載入）
API Key 外洩 資料被未授權讀取 Key 只存 server-side env、加 rate limit、只開放 GET
大量顧客同步時效能 API 被打爆 批量同步加入 queue + rate limit（每秒最多 10 筆）
兩邊 profile 資料不一致（名字/電話不同） 店長困惑 只顯示身體數據、不覆蓋蒸足系統資料。明確標示「來自身體數據系統」
